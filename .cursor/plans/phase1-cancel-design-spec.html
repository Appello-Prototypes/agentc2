Backend
API Routes

Route 1: POST /api/workflows/[slug]/runs/[runId]/cancel

Aspect Detail
Auth authenticateRequest() -- session or API key
Guard Verify workflow belongs to org. Verify run status is QUEUED, RUNNING, or has suspendedStep
Mutation Set status → CANCELLED, completedAt → now(), clear suspendedStep, suspendedAt, suspendDataJson
Side effects Reject all linked ApprovalRequest records (status=pending → rejected). Refresh workflow metrics.
Response 200 { success: true, cancelled: true, run: { id, status, completedAt } }
Response 409 { success: false, error: "Run is not in a cancellable state" }

Route 2: POST /api/reviews/[id]/cancel

Aspect Detail
Purpose Convenience wrapper -- resolves review → workflow slug + run ID → performs cancel inline
Auth authenticateRequest()
Guard Verify review exists and belongs to org, has workflowRunId
Logic Looks up ApprovalRequest → WorkflowRun → Workflow. Performs same cancel mutations inline.
Response Same shape as Route 1

Edge Cases
Behavioral Details
Scenario Behavior
Run already completed/failed API returns 409 Conflict. Toast: "Run has already finished"
Run already cancelled API returns 409. Toast: "Run was already cancelled"
No linked workflow run Cancel button not rendered (guarded by review.runId)
Race with approve/reject API rejects gracefully if review already resolved
Network error Toast: "Failed to cancel run". Button re-enabled, confirmation dismissed.
Cancel after partial execution Completed steps preserved. Expanded detail panels remain viewable.
Expand step while confirming cancel Step detail expands normally -- cancel confirmation stays visible below stepper
Card after cancel Card removed from pending list
