Phase 3

Skip and Continue

Give operators an escape hatch for blocked workflows. When a step is suspended (awaiting human approval) or failed, a "Skip & Continue" button bypasses it by injecting a synthetic output and resuming the workflow from the next step. Builds on Phase 0 (expandable step details), Phase 1 (cancel), and Phase 2 (retry).

Architecture
Skip Flow (Dual Path)
Skip handles two distinct scenarios: suspended steps (human approval pending) and failed steps. Each follows a different backend path but produces the same user-facing result -- the step is marked as skipped and the workflow continues.

Two paths, one result: For suspended steps, skip calls the existing resume API with { skipped: true, reason } -- the human step handler receives this as its output and the workflow continues naturally. For failed steps, skip injects a synthetic output into existingSteps and re-executes the workflow, effectively bypassing the failed step.

Implementation
Step-by-Step

Step 1: Skip API Route

File: apps/agent/src/app/api/reviews/[id]/skip/route.ts Pattern: apps/agent/src/app/api/reviews/[id]/retry/route.ts (Phase 2)
Key Imports
import { prisma, Prisma } from "@repo/database"
import { executeWorkflowDefinition, type WorkflowDefinition } from "@repo/agentc2/workflows"
import { refreshWorkflowMetrics } from "@/lib/metrics"
import { authenticateRequest } from "@/lib/api-auth"
Logic
  1. Authenticate request via authenticateRequest()
  2. Parse { stepId, reason? } from request body
  3. Look up ApprovalRequest by id, include workflowRun with steps and workflow
  4. Validate: approval exists, has a linked workflowRun
  5. Find the target step -- confirm its status is FAILED or the run's suspendedStep matches stepId
  6. SUSPENDED PATH (run.suspendedStep === stepId):
    1. Build resume data: { decision: "skipped", skipped: true, approved: false, reason, skippedBy, skippedAt, channel: "command-ui" }
    2. Call the resume API internally: POST /api/workflows/{slug}/runs/{runId}/resume with the skip resume data
    3. Update the ApprovalRequest status to "skipped" with decisionReason
  7. FAILED PATH (step.status === "FAILED"):
    1. Build existingSteps from all COMPLETED steps
    2. Inject synthetic output for the failed step: existingSteps[stepId] = { skipped: true, reason, skippedBy, skippedAt, originalError }
    3. Set run status to RUNNING
    4. Call executeWorkflowDefinition({ definition, input, existingSteps }) (no resume -- all steps including the skipped one are in existingSteps)
    5. Update the failed step's DB record: status COMPLETED, outputJson = synthetic output
  8. Update remaining WorkflowRunStep records with execution results
  9. Update WorkflowRun final status
  10. Call refreshWorkflowMetrics()
  11. Return { success: true, status, runId, skippedStepId, path }
Error Responses
Code Condition
401 Unauthenticated
404 Approval request not found
400 No workflow run linked to this review
400 stepId not provided
409 Step is COMPLETED -- cannot skip a completed step
409 Step is PENDING -- not yet reached, cannot skip
409 Run is in terminal state (COMPLETED, CANCELLED)
500 Resume or executeWorkflowDefinition failed

Logic
  1. Inside the expanded step detail, check if step is FAILED or SUSPENDED
  2. If skippable and onSkipStep is provided, render "Skip & Continue" button
  3. For FAILED steps: render alongside the Retry button (Phase 2) as a secondary action
  4. For SUSPENDED steps: render as a standalone button below the decision prompt
  5. On click: set confirmSkipStepId to show confirmation with optional reason textarea
  6. Confirmation includes a warning that downstream steps may receive incomplete data (for failed path)
  7. On confirm: set skipStepId, call onSkipStep(step.stepId, skipReason || undefined)
  8. On success: clear states, re-fetch steps
  9. After skip: the step visually transitions to a "skipped" state with a slate-colored dot and skip icon
Edge Cases
Failure Scenarios
Scenario Backend Frontend
Step is COMPLETED (race condition) Return 409: "Cannot skip a completed step" Show error toast, re-fetch steps
Step is PENDING (not yet reached) Return 409: "Step not yet reached" Skip button not rendered for PENDING steps (pre-validated)
Run was cancelled while skip in progress Return 409: "Run is cancelled" Show error toast, re-fetch to update stepper
Skip succeeds but next step also fails Return { status: "failed" } Re-fetch: skipped step shows as completed/skipped, next step shows failed with its own retry/skip
Skip succeeds and next step suspends Return { status: "suspended" } Stepper updates: skipped step completed, next step suspended
Downstream step relies on skipped step's output Step receives synthetic { skipped: true } as input; may fail if it expects specific fields Warning in confirmation: "Downstream steps may receive incomplete data"
User provides very long reason text Stored in DB as-is (text field) Textarea has resize: vertical; no character limit enforced
Empty reason Defaults to "User skipped" Reason field is optional; placeholder hints at expected content
Skip on last step in workflow Workflow completes immediately after skip Stepper shows all steps complete (last one as skipped)
Network timeout during skip N/A (client-side) Show error toast "Network error", clear loading state
Approval already resolved (e.g., approved in Slack while skipping in UI) Return 400 via existing status check Show error toast, card may be removed from list

Data Flow
Call Chain
User clicks "Skip & Continue" on suspended/failed step
    └─ WorkflowStepsCard: setConfirmSkipStepId(stepId)
User enters optional reason, clicks "Yes, Skip"
    └─ WorkflowStepsCard: setSkipStepId(stepId)
        └─ WorkflowStepsCard: onSkipStep(stepId, reason)
            └─ DecisionCard: onSkipStep(review.id, stepId, reason)
                └─ page.tsx: handleSkipStep(reviewId, stepId, reason)
                    └─ useReviews.handleSkipStep()
                        └─ POST /api/reviews/{id}/skip { stepId, reason }
                            └─ authenticateRequest()
                            └─ prisma.approvalRequest.findUnique(...)
                            └─ Determine path: suspended vs. failed
                            ├─ SUSPENDED:
                            │   └─ POST /api/workflows/{slug}/runs/{runId}/resume
                            │       └─ resumeData: { skipped: true, reason, ... }
                            │   └─ prisma.approvalRequest.update({ status: "skipped" })
                            └─ FAILED:
                                └─ Build existingSteps from COMPLETED + synthetic for failed
                                └─ executeWorkflowDefinition({ existingSteps })
                                └─ prisma.workflowRunStep.update (failed → COMPLETED)
                            └─ Upsert new step records
                            └─ Update WorkflowRun status
                            └─ refreshWorkflowMetrics()
                        └─ On success: addToast("Step skipped, workflow continuing")
                        └─ fetchReviews() + fetchMetrics()
