Concept 5 of 5

Topology Flow

A living system map showing the actual workflow topology as a directed graph. Nodes represent workflow steps, edges represent transitions, and items are dots flowing through the graph. Bottlenecks are visible as count badges accumulating at specific nodes. The operator sees the entire system at once and can batch-act on all items at any node.

Data Flow
Workflow topology is derived from the Workflow.definitionJson (step graph). Active items are mapped to their current node position. The DAG renderer places nodes in a left-to-right layered layout.

Step 1: Topology API Route

File: apps/agent/src/app/api/workflows/topology/route.ts
Interface
// GET /api/workflows/topology
// Returns workflow step graphs for DAG rendering

interface TopologyResponse {
  workflows: WorkflowTopology[]
}

interface WorkflowTopology {
  slug: string
  name: string
  activeCount: number
  steps: TopologyStep[]
  edges: TopologyEdge[]
}

interface TopologyStep {
  id: string
  name: string
  type: string  // "tool" | "agent" | "branch" | "dowhile" | "human"
  isGate: boolean
}

interface TopologyEdge {
  from: string
  to: string
  label?: string  // branch condition
}
Logic
  1. Query all active workflows for the workspace
  2. Parse definitionJson to extract step IDs, names, types, and transitions
  3. Identify gate steps by matching patterns (suspend, human, approval keywords)
  4. Count active runs per workflow from WorkflowRun where status != COMPLETED/CANCELLED
  5. Return topology for each workflow with active count
Code Condition
401 Unauthenticated
403 Workspace mismatch

Step 2: useTopologyMap Hook

File: apps/agent/src/app/command/hooks/useTopologyMap.ts
Interface
interface TopologyMap {
  selectedWorkflow: WorkflowTopology | null
  nodeItems: Map<string, ReviewItem[]>  // stepId -> items at that node
  nodeCounts: Map<string, { total: number; waiting: number; failed: number }>
  totalGateItems: number
}

function useTopologyMap(
  topologies: WorkflowTopology[],
  reviews: ReviewItem[],
  stepCache: Map<string, StepData[]>,
  selectedSlug: string
): TopologyMap
Logic
  1. Filter reviews by selectedSlug's workflowSlug
  2. For each review, find its current step from stepCache (last non-QUEUED step)
  3. Map review to the corresponding topology node by stepId matching
  4. Compute per-node counts (total, waiting, failed)
  5. totalGateItems = sum of waiting items at gate nodes

Step 3: TopologyNode Component

File: apps/agent/src/app/command/components/TopologyNode.tsx
Props
interface TopologyNodeProps {
  step: TopologyStep
  counts: { total: number; waiting: number; failed: number }
  isSelected: boolean
  onClick: (stepId: string) => void
}
Logic
  1. Gate nodes: amber border, amber text, pulse animation on badge when waiting > 0
  2. Regular nodes: default border, blue badge when items > 0
  3. Failed badge: red when any items are failed at this node
  4. Badge hidden when count is 0
  5. Click opens NodeDrawer with all items at this step

Step 4: NodeDrawer Component

File: apps/agent/src/app/command/components/NodeDrawer.tsx
Logic
  1. Slides up from bottom when a node is clicked
  2. Header: step name, waiting count, "Approve All" batch button
  3. List of items at this step with: risk dot, ID, title, age, individual approve/reject buttons
  4. Close on Escape or clicking overlay
  5. After batch approve: all items removed, node count decrements, drawer closes

Step 5: TopologyFlowView + Page Integration

File: apps/agent/src/app/command/components/TopologyFlowView.tsx
Logic
  1. Workflow selector pills at top (one per workflow with active count)
  2. DAG layout: vertical columns for main path, horizontal branches for parallel steps
  3. Edges rendered as CSS lines (or SVG for complex branching)
  4. Legend at bottom explaining node colors and badge meanings
  5. Integrates into page.tsx as "Topology" view toggle
Edge Cases
Boundary Conditions
Scenario Backend Frontend
0 active workflows Empty topology array Map icon with "No active flows" message
Workflow with 50+ steps Full definitionJson Group related steps into collapsed "zones"; scrollable DAG with mini-map
Complex branching (5+ paths) Multiple edges from one node Horizontal layout for branches, labeled edges for conditions
Node with 20+ items All items returned Scrollable drawer list; "Approve All" handles batch
Step not in topology definitionJson may differ from run Unmapped items shown in "Other" floating node
Workflow definition changes mid-run Old definitionJson in run Use run's compiled steps, not current workflow definition

Data Flow
Call Chain
User opens Command with "Topology" view
    └─ page.tsx: viewMode === "topology"
        └─ GET /api/workflows/topology → workflow step graphs
        └─ useReviews() → all reviews + step prefetch
        └─ useTopologyMap(topologies, reviews, stepCache, selectedSlug)
            └─ Maps items to nodes, computes counts per node
        └─ <TopologyFlowView />
            └─ Workflow selector pills
            └─ DAG with TopologyNode components
            └─ Count badges on nodes, pulse on gates

User clicks "Merge Review" gate node
    └─ TopologyFlowView: setSelectedNode("merge-review")
        └─ NodeDrawer opens from bottom
            └─ Shows 3 items: #187, #202, #190
            └─ Individual approve/reject per item
            └─ "Approve All (3)" batch button

User clicks "Approve All"
    └─ NodeDrawer: batchApprove(itemIds)
        └─ useReviews().handleBatchApprove(ids)
            └─ POST /api/reviews/[id]/approve for each
            └─ Node badge count → 0, drawer closes
