Phase 2

Retry Failed Step

When a workflow step fails (e.g., GitHub API 401, agent timeout), let operators retry that specific step without re-running the entire workflow. Preserves all completed step outputs via existingSteps and re-invokes executeWorkflowDefinition targeting the failed step. Builds on Phase 0 (expandable step details) and Phase 1 (stepper + cancel).

Architecture
Retry Flow
The retry action follows the same pattern as the existing resume route -- it collects completed step outputs, then re-executes from the failed step. A convenience route scoped to the review ID delegates to the workflow resume API.

Implementation
Step-by-Step

Step 1: Retry API Route

File: apps/agent/src/app/api/reviews/[id]/retry/route.ts Pattern: apps/agent/src/app/api/workflows/[slug]/runs/[runId]/resume/route.ts
Key Imports
import { prisma, Prisma } from "@repo/database"
import { executeWorkflowDefinition, type WorkflowDefinition } from "@repo/agentc2/workflows"
import { refreshWorkflowMetrics } from "@/lib/metrics"
import { authenticateRequest } from "@/lib/api-auth"
Logic
  1. Authenticate request via authenticateRequest()
  2. Parse { stepId } from request body
  3. Look up ApprovalRequest by id param, include workflowRun.steps and workflowRun.workflow
  4. Validate: approval exists, has a linked workflowRun, and the run is in RUNNING or FAILED status
  5. Find the target step in workflowRun.steps -- confirm its status is FAILED
  6. Build existingSteps from all steps with status COMPLETED (exclude the failed step)
  7. Set the run status to RUNNING before re-execution
  8. Call executeWorkflowDefinition({ definition, input: run.inputJson, resume: { stepId, data: {} }, existingSteps, workflowMeta })
  9. Update WorkflowRunStep records with new results (same upsert logic as resume route)
  10. Update WorkflowRun status based on result (COMPLETED, FAILED, or remain RUNNING if suspended)
  11. Call refreshWorkflowMetrics()
  12. Return { success: true, status, runId, retriedStepId }
Error Responses
Code Condition
401 Unauthenticated
404 Approval request not found
400 No workflow run linked to this review
409 Target step is not in FAILED state
409 Run is in a terminal state (COMPLETED, CANCELLED)
500 executeWorkflowDefinition threw

Step 2: useReviews Handler

File: apps/agent/src/app/command/hooks/useReviews.ts
Interface
async function handleRetryStep(reviewId: string, stepId: string): Promise<void>
Logic
  1. Set actingId to reviewId
  2. POST to ${getApiBase()}/api/reviews/${reviewId}/retry with { stepId }
  3. On success: addToast("Step retried successfully"), re-fetch reviews and metrics
  4. On failure: addToast(data.error, "error")
  5. Finally: clear actingId
Export
return {
  ...existingExports,
  handleRetryStep
}

Step 4: Retry UI in WorkflowStepsCard

File: apps/agent/src/app/command/components/WorkflowStepsCard.tsx
New State
const [retryingStepId, setRetryingStepId] = useState<string | null>(null)
const [confirmRetryStepId, setConfirmRetryStepId] = useState<string | null>(null)
Logic
  1. Inside the expanded step detail panel (currently showing output/error), check if step.status === "FAILED"
  2. If failed and onRetryStep is provided, render "Retry Step" button below the error block
  3. On click: set confirmRetryStepId to show confirmation bar (reuse Phase 1 confirm pattern)
  4. On confirm: set retryingStepId, call onRetryStep(step.stepId), then re-fetch steps
  5. On success: clear states, show updated step data from re-fetch
  6. Confirmation bar text: "The step {name} will be re-executed. All prior completed steps will be preserved."
Edge Cases
Failure Scenarios
Scenario Backend Frontend
Step is no longer FAILED (race condition) Return 409 with current status Show error toast: "Step is no longer in a failed state"
Run is CANCELLED while retry in progress Return 409: "Run is cancelled" Show error toast, re-fetch to update stepper
Retry succeeds but next step suspends Return { status: "suspended" } Re-fetch steps; stepper shows retried step as completed, next step as suspended
Retry fails again Return { status: "failed" } Re-fetch steps; step still shows FAILED with new error, retry button reappears
Network timeout during retry N/A (client-side) Show error toast "Network error", clear loading state, retry button remains
No onRetryStep callback provided N/A Retry button not rendered (graceful degradation)
Multiple failed steps in one run Only the specified stepId is retried Each failed step independently shows its own retry button
Underlying issue not resolved (e.g., token still expired) executeWorkflowDefinition fails again, same error Step shows FAILED again with updated error; user can retry again
Approval request already resolved Return 400: "Review already resolved" Show error toast, card may have been removed from list

Component Architecture
Component / Function Purpose
POST /api/reviews/[id]/retry Convenience route: resolves approval → workflowRun → validates failed step → calls executeWorkflowDefinition with existingSteps
handleRetryStep() Hook handler: POST to retry API, toast, re-fetch reviews + metrics

Data Flow
Call Chain
User clicks "Retry Step" on failed step
    └─ WorkflowStepsCard: setConfirmRetryStepId(stepId)
User clicks "Yes, Retry"
    └─ WorkflowStepsCard: setRetryingStepId(stepId)
        └─ WorkflowStepsCard: onRetryStep(stepId)
            └─ DecisionCard: onRetryStep(review.id, stepId)
                └─ page.tsx: handleRetryStep(reviewId, stepId)
                    └─ useReviews.handleRetryStep()
                        └─ POST /api/reviews/{id}/retry { stepId }
                            └─ authenticateRequest()
                            └─ prisma.approvalRequest.findUnique({ include: workflowRun })
                            └─ Validate: step.status === "FAILED"
                            └─ Build existingSteps from COMPLETED steps
                            └─ prisma.workflowRun.update({ status: "RUNNING" })
                            └─ executeWorkflowDefinition({ resume: { stepId, data: {} }, existingSteps })
                            └─ Upsert WorkflowRunStep records
                            └─ Update WorkflowRun final status
                            └─ refreshWorkflowMetrics()
                        └─ On success: addToast("Step retried successfully")
                        └─ fetchReviews() + fetchMetrics()
