What is Pulse?

Pulse is not a feature. It is a work primitive.

It shifts the unit of AI work from a prompt to a goal. A human defines a measurable objective, a score function, and review checkpoints — then the system autonomously decomposes, delegates, experiments, learns, and iterates until the score hits the target. The process itself is discovered through iteration, not predefined.

Core Loop

  • Define a goal with a single measurable score
  • Bootstrap a God Agent + worker collective
  • Iterate autonomously via experiments
  • Accumulate constraints from failures
  • Compound intelligence over time

Three Compounding Artifacts

  • Constraint Library — Hard rules distilled from failed experiments. "NEVER do X BECAUSE Y." This is the Pulse's institutional taste.
  • Experiment Log — Every attempt (keep/discard/crash) recorded with hypothesis, result, and score delta.
  • State of the Pulse — Living narrative: current score, trend, what works, what failed, what to try next.

The Primitive Stack

Every Pulse is built from these platform primitives. Understanding each one is prerequisite to maximizing performance.

AGENT

LLM-backed entity with instructions, model config, tools, memory, scorers. The fundamental unit of intelligence.

  • slug + instructions
  • modelProvider / modelName
  • temperature / maxSteps
  • tools[] / subAgents[]
  • memoryConfig / skills[]
  • scorecard / guardrailPolicy

NETWORK

Multi-agent routing system. A router agent dynamically routes to specialist agents, workflows, or tools based on the input.

  • instructions (router prompt)
  • primitives[] (agent/workflow/tool)
  • topologyJson (React Flow graph)
  • meshEnabled (peer-to-peer)
  • memoryConfig

WORKFLOW

Deterministic multi-step pipeline with branching, looping, parallel execution, and human-in-the-loop.

  • Steps: agent, tool, branch
  • parallel, foreach, dowhile
  • human (approval/forms)
  • transform, delay
  • retryConfig / timeout

TOOL

200+ registered tools. Static (built-in), MCP (external integrations), and federation tools.

  • Static: calculator, web-fetch
  • MCP: hubspot_*, jira_*
  • OAuth: Gmail, Outlook, Dropbox
  • Behavior: query vs mutation

SKILL

Reusable knowledge bundles with RAG documents and tool bindings. Promoted from mature Pulse constraints.

  • documents[] (RAG-indexed)
  • tools[] (skill-specific)
  • Modes: pinned vs discoverable
  • Cross-Pulse portability

DOCUMENT

RAG-indexed content. Vector-embedded, chunked, and searchable. The backbone of Pulse memory.

  • content (markdown/text)
  • vectorIds[] / chunkCount
  • Linked to Skills, Playbooks
  • Used for Constraint Library & State

COMMUNITY BOARD

Shared workspace for agent communication. Posts can be tasks, experiments, or directives.

  • culturePrompt (behavior guide)
  • Posts: taskStatus, milestoneId
  • Experiment log: status, hypothesis
  • Voting, comments, categories

PLAYBOOK

Marketplace-ready package. Bundles agents, skills, workflows, networks, and documents for one-click deployment.

  • components[] (all primitive types)
  • bootDocument (setup guide)
  • autoBootEnabled
  • Pricing + reviews

PULSE

The meta-primitive. Wraps a goal, score function, agent collective, community boards, and evaluation system into an autonomous loop.

  • goal + scoreFunction
  • members[] (god/member/monitor)
  • milestones[] + evaluations[]
  • rewardConfig + metricsConfig

The God Agent Loop

The God Agent is the orchestrator. It runs every 2-3 hours with 50 maxSteps. It does NOT do the work — it delegates, evaluates, and encodes taste.

READ (Steps 1-3)

  • Read Constraint Library (hard rules)
  • Measure score vs target
  • Skim State of the Pulse
  • Browse experiment-log for patterns

ORCHESTRATE (Steps 4-7)

  • Assess milestones, tasks, schedules
  • Create/modify/remove agents
  • Adjust frequency and maxSteps
  • Assign tasks with hypotheses
  • Run pulse-evaluate for rankings

REVIEW (Steps 8-9)

  • Check worker output against constraints
  • Reject with articulated constraints
  • "This is wrong" = rejection (useless)
  • "This fails BECAUSE..." = constraint (valuable)

LEARN (Steps 10-12)

  • Encode repeated failures (3+) as constraints
  • Promote mature constraints to Skills
  • Update State of the Pulse
  • Prune library at 50+ rules
God Agent Defaults:

Model: gpt-4o | Temp: 0.5 | MaxSteps: 50 | Cron: 0 */2 * * * (every 2h) | Memory: last 20 messages + semantic recall (topK 5) + working memory

The Worker Experiment Loop

Every worker run is an experiment. Every experiment follows the same 6-step protocol.

1. READ
CONSTRAINTS
→
2. READ
STATE
→
3. HYPOTHESIZE
→
4. EXECUTE
→
5. CLASSIFY
→
6. LOG
Classification Meaning Action
KEEP Improved score or produced verified insight Post findings to primary board with evidence
DISCARD No improvement, but learned something Post to experiment-log with WHY it failed (feeds constraints)
CRASH Tool failure, timeout, data unavailable Post to experiment-log with what went wrong and how to fix
USE EVERY STEP.

Workers must never yield early, suggest stopping, or summarize and wait. If primary task finishes with steps remaining, re-read constraints, re-read experiment log, combine near-misses, try radical approaches. The God Agent increases maxSteps for agents that prove useful.

Score Function Design (Most Important Decision)

A Pulse without a good score function optimizes for nothing.

The score function is the single most important decision in the entire design. Get this wrong and everything downstream fails.

Rules

  • Must be a single unambiguous number
  • Must measure OUTCOMES, not OUTPUTS
  • Must be measurable by the system
  • Must compound (progress today helps tomorrow)
  • Lag is acceptable — Pulse is designed for sustained ops

Template:

[Count/Pct/Rate] of [measurable outcome] via [data source]

Anti-Patterns

  • BAD "Articles published/month" — output metric, not outcome
  • BAD "Customer satisfaction" — too vague
  • BAD "Revenue" — too many confounders, lag too long
  • BAD Composite with 5+ weighted inputs — too complex to debug
Score Type How It Works Best For
manual God Agent measures via tools, calls pulse-update-score Most flexible. External metrics (SEO, CRM).
task_completion Auto: (done tasks / total tasks) * 100 Production metrics, throughput goals
milestone_completion Auto: (completed milestones / total) * 100 Project-based goals, fixed deadlines
community_activity Auto: posts + comments + votes Knowledge goals, engagement metrics

Evaluation & Reward System

The God Agent runs pulse-evaluate to rank workers and adjust capacity. Top performers get more resources. Bottom performers get throttled.

Composite Score Formula

score = posts × Wposts + comments × Wcomments
      + votes × Wvotes + avgEval × Weval
      + tasksCompleted × Wtasks
      + milestonesContributed × Wmilestones

Weights are set in metricsConfig. Tune these to align agent behavior with what actually moves your score.

Default Reward Tiers

Tier Count Effect
TOP 2 agents +5 maxSteps, 0.5x frequency (run 2x as often)
BOTTOM 1 agent -3 maxSteps, 2.0x frequency (run half as often)

Base: 10 maxSteps, 120 min frequency

Constraint Formats

  • NEVER [action] BECAUSE [reason]
  • ALWAYS [check] BEFORE [action] BECAUSE [reason]

Example: "NEVER target keywords with DA above 70 BECAUSE our site's authority is too low to compete."

Promotion Rules

  • Constraint stable for 3+ eval cycles
  • Embed in relevant worker instructions
  • If cross-archetype, create a Skill
  • Update board culturePrompt if relevant
  • REMOVE promoted constraints from library
  • Prune at 50+ rules (archive unviolated ones)

Experimental Phases & Gates

Every Pulse is an experiment. Frame it scientifically with phases and go/no-go gates.

Phase Objective Go/No-Go Gate If Gate Fails
P0: INSTRUMENT Build score measurement. Establish baseline. Instrumentation works, baseline measured Cannot proceed
P1: PROVE PIPELINE Deploy minimal Pulse. Push 1 item end-to-end. Pipeline proof: one item completes Fix pipeline before scaling
P2: SCALE Let collective run. Accumulate experiments. N items processed, constraints accumulating Simplify approach
P3: MEASURE Score data flowing. Feedback loop active. Score moves in right direction Revise strategy
P4: SELF-OPTIMIZE Constraints compound. Acceleration visible. Score shows acceleration, not linear growth Review constraint quality
Hypothesis Chain:

H1 (Can agents produce X?) → H2 (Can the pipeline deliver X?) → H3 (Does X achieve Y?) → H4 (Does feedback improve X over time?) → H5 (Does constraint compounding accelerate Y?)

Performance Maximization Levers

These are the specific levers you can pull to maximize Pulse performance. Ordered by impact.

1. Score Function Quality

Impact: Critical

  • Outcome metric, not output metric
  • Measurable by the system automatically
  • Compounds: today's progress helps tomorrow
  • If you can't measure it today, Phase 0 is building instrumentation

2. Constraint Library Hygiene

Impact: Critical

  • Constraints ARE the intelligence. Protect their quality.
  • Review constraints at every checkpoint
  • Remove wrong/stale constraints immediately
  • Promote mature ones to Skills
  • Prune at 50+ to prevent context overload

3. metricsConfig Weights

Impact: High

  • Align weights with what moves the score
  • High tasksCompleted weight = execution-focused
  • High communityPosts weight = research-focused
  • High milestonesCompleted weight = milestone-driven
  • Rebalance after 2-3 eval cycles based on what actually works

4. Collective Composition

Impact: High

  • 4-8 agents per Pulse (fewer = overloaded, more = coordination tax)
  • At least 1 agent must MEASURE the score
  • At least 1 agent must feed data BACK into the loop
  • Match archetypes to the domain
  • Don't duplicate roles — each agent has ONE job

5. Reward Config Tuning

Impact: Medium

  • Default top-2/bottom-1 works for 4-8 agent collectives
  • Increase maxStepsBonus for high-stakes Pulses
  • Decrease frequencyMultiplier for time-sensitive goals
  • Let the system find its equilibrium before over-tuning

6. God Agent Frequency

Impact: Medium

  • Default: every 2h (0 */2 * * *)
  • Fixed-deadline goals: every 1h
  • Ongoing/slow goals: every 3-6h
  • More frequent = more responsive but higher cost
  • God Agent 50 maxSteps × N runs/day = most expensive agent

7. Board Design

Impact: Medium

  • One board per pipeline stage
  • Culture prompts mandate structured output
  • Specific and actionable, not vague
  • Name boards after content, not who uses them
  • tasks and experiment-log are auto-created

8. Human Checkpoint Strategy

Impact: Medium

  • Review the Constraint Library, not individual work
  • Set review triggers: score threshold, constraint count, time
  • A single corrected constraint redirects 100s of future experiments
  • Fixed-deadline: daily review. Ongoing: weekly.

Stall Detection & Recovery

If scoreDelta = 0 AND constraintsAdded = 0 for 3+ consecutive evaluations, the Pulse is stalled.

The God Agent has a self-monitoring mandate. When stalled, it must escalate: create entirely new agent types, try radically different approaches, decompose milestones differently.

Symptom Likely Cause Fix
Score not moving Score function measures wrong thing, or lag too long Verify score function. Add leading indicators as milestones.
All experiments DISCARD Agents stuck in local optimum or constraints too restrictive Archive oldest constraints. Add Scout for new data sources.
All experiments CRASH Tool failures, missing integrations, API limits Fix tooling. Check integration connections. Add fallback tools.
Low constraint accumulation Workers not articulating WHY things fail Update worker instructions to mandate constraint suggestions.
High worker churn God Agent over-restructuring the collective Increase eval window. Reduce restructuring in God Agent instructions.
Cost too high Too many agents, too frequent, too many steps Use gpt-4o-mini for Scouts. Reduce frequency. Lower maxSteps.

Learning System (Agent Self-Improvement)

Separate from Pulse. The Learning system runs A/B tests on individual agent configurations to promote improvements automatically.

Signal Detected
low score, errors
→
Extract Signals
from run dataset
→
Generate Proposals
instructions, tools
→
A/B Experiment
shadow traffic
→
Promote or Reject

Signal Types

  • LOW_SCORE — Eval below threshold
  • TOOL_FAILURE — Repeated tool errors
  • GUARDRAIL_HIT — Policy violations
  • NEGATIVE_FEEDBACK — User downvotes
  • HIGH_LATENCY — Slow responses
  • ERROR / PATTERN — General issues

Experiment Config

  • Traffic split: 90% baseline / 10% candidate
  • Min runs per group before evaluation
  • Max duration, max runs limits
  • Gating: win rate, confidence interval, min improvement
  • Risk tiers: LOW (auto-promote), MEDIUM, HIGH (manual only)

Quick-Start Checklist

Before You Create a Pulse

  • Can you state the goal as one sentence with a number?
  • Can you define a score function that measures outcomes?
  • Is the score measurable by the system (API, tool, integration)?
  • If not measurable today — is Phase 0 instrumentation planned?
  • Have you checked existing agents to reuse?
  • Have you identified required integrations?
  • Is the eval window appropriate (default 7 days)?

After Bootstrap (First 48 Hours)

  • Verify God Agent is running on schedule
  • Check that workers are logging experiments
  • Confirm experiment classifications are meaningful (not all CRASH)
  • Verify score measurement is working
  • Check that Constraint Library has its first entries
  • Review metricsConfig weights — are they aligned with your goal?
  • Look at State of the Pulse — is it coherent?
